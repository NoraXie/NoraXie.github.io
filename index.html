Jenkins项目构建发布指南

发表于 2017-02-27 |

内容摘要

  1. 部署服务器环境
  2. 登录jenkins
  3. new item 创建项目
  4. build now 构建项目

部署服务器环境

开发环境选择d1-my-dev-001这台服务器, 注意 在这台主机上新建的目录,如果需要远程主机访问,都应该是wosai属主和属组,否则在构建过程中执行脚本会遇到Permission Denied

  1. 通过跳板机登录到d1-my-dev-001主机,选择superuser权限
  2. sudo vi /app/bin/nginx/conf/sites-enabled/project-name.conf

2.sudo vi /etc/supervisor/conf.d/project-name.conf

  1. sudo mkdir /app/bin/project-name
  2. sudo mkdir /app/log/project-name
  3. sudo cp -r /app/bin/任意项目/jetty /app/bin/project-name/
  4. sudo chown wosai:wosai /app/bin/project-name
  5. sudo nginx -t ## 检查一下nginx配置是否有域名冲突的问题
  6. sudo nginx -s reload ##重新加载nginx服务,使配置文件生效
  7. sudo supervisorctl update project-name ## linux进程管理器更新配置后的进程使之生效

登录Jenkins

通过王伟提供的Jenkins账号,登录Jenkins.
在选线卡中选择需要发布项目的主机,如 Develop ENV

创建项目

选中主机后,在页面左上角有New Item选项,点击后开始创建项目.
1.输入项目名称
2.如果选择maven工具构建项目,则选择maven project
2.1 为该项目选择源码仓库地址,一般放在gitlab中,输入https的地址即可
2.2 为该项目设置构建生命周期,一般构建时的步骤是清空,编译,安装并发布,用 clean -u install deploy 命令
2.3 设置项目发布后执行脚本Execute Shell,该脚本负责将构建好的war包发送到开发主机上相应的目录,如wosai@d1-my-dev-001:/app/bin/redpacket-lib/jetty/webapps/redpacket-lib.war
发送成功后远程启动该项目
如ssh -t -t wosai@d1-my-dev-001 ‘sudo supervisorctl start redpacket-lib’

总结

分布式事务

场景
一台MySQL，一台Redis，两台应用服务器，用户的数据存储持久化在MySQL中，缓存在Redis，有请求的时候从Redis中获取缓存的用户数据，有修改则同时修改MySQL和Redis中的数据。现在问题是：1.先保存到MySQL和先保存到Redis都面临着一个保存成功而另外一个保存失败的情况，这样，如何保证MySQL与Redis中的数据同步？2.两台应用服务器的并发访问，如何保证数据的安全性？

读Redis，写MySQL，同时消费MySQL binlog更新Redis
MySQL binlog增量订阅消费+消息队列+处理并把数据更新到redis

说个大概吧，我们热数据基本都是redis，增删改都是操作mysql，对于读是保存到redis，这样就涉及到数据同步操作，同步操作分为两大块，我们的叫法是，一个是全量(将全部数据一次写入到redis，时间几小时不等)，一个是增量（实时更新）。这里说的是增量，主要问题是即时性，因为增删改都是直接操作mysql变更都在MySQL（这里高并发的问题是用分库分表加外层的负载均衡） 所以我们的方向是读取binlog然后分析 ，利用消息推送到某服务器A，再进行分析，然后更新各台redis，消息推送工具用的是rabbitMQ，可设定某表的变更推送(分三类update insert delate 包含变更前后的数据)，这里有个问题是：mysql数据操作太频繁产生的推送可能会很多，所以分析处理脚本处理速度一定要跟得上（我用Python写，前期多线程（坑），后来改成多进程），还有一个问题是，对于mysql-redis的数据关系映射设定不要太复杂，一表对一表就行，数据组合交给业务层做，这样分析处理脚本不会太多负担，处理速度更快，而且操作redis也更简单，redis每个对应mysql数据表的可使用多端口多实例，redis是单线程而且这样对于redis的主从和负载均衡有利.对于服务器A 可以再给其它服务做一个数据表增量变更数据获取接口，利用数据纬度，获取时间段的变更数据。追加，对于订单类部分，都是完全使用mysql，这个做好数据服务器，DB，table，分区，的拆分就好了，看并发请求越多拆分越多。上面说太多都是屁话，其实就是MySQL binlog增量订阅消费+消息队列+处理并把数据更新到redis

QPS与并发数

系统吞度量要素：一个系统的吞度量（承压能力）与request对CPU的消耗、外部接口、IO等等紧密关联。单个reqeust 对CPU消耗越高，外部系统接口、IO影响速度越慢，系统吞吐能力越低，反之越高。
系统吞吐量几个重要参数：QPS（TPS）、并发数、响应时间
QPS（TPS）：每秒钟request/事务 数量
并发数： 系统同时处理的request/事务数
响应时间： 一般取平均响应时间
