CRM项目乱相

发表于 2017-03-19 | 分类于 工作 |

CRM

Spring boot properties

Common properties

商户密码的背景

1.0中存有商户密码,但是2.0没有商户密码.
而在2.0之后并没有使用过1.0的商户密码.
但是现在2.0需要先查询有没有商户密码,如果有则直接写入;没有则需要先获取商户密码后写入

解决方案

CRM商户入网

注册一个商户

入网成功后

CRM需要调用app-backend的接口生成一个商户密码,之后app就可以获取商户密码了.然后调短信提醒接口将商户信息发送给这个商户的手机短信模板是亦天写的,所以关于短信模板需要咨询亦天

对公银行的背景

银行私人账号:用户在前端输入银行卡号后可以直接获取开户行信息
对公账号:无法根据银行卡号获取相关的开户行

解决方案

调用app-backend的rpc/backend接口,获取对公开户行,一共有175条
直接在前端显示太长,所以前端最好提供一个输入框,可以根据关键字查询到相关的银行.

商户入网的历史问题

商户入网仍然依赖于1.0的channel_code,一个channel_code代表一个BD.channel_code这个字段在1.0的channel表中.
1.0(核心业务系统和支付网关)中只存了store(门店)
2.0(核心业务系统和支付网关)中存有merchant(商户),store(门店)

CRM主要管理Merchant与BD,BD与BD之间的关系
CRM将这几个的关系拆分成了organization,user,role.
organization维护着BD们的关系
user维护单个user的关系
role维护user的权限

1.0channel表 –> CRM

这个过程依赖于长茂的dts,相当于监控1.0的store表,之后王浩将dts的同步数据拆分成organization,user,role放在mongodb中

2.0upay-core –> CRM

organization,user,role 与 channel_code一一对应,而channel_code与store有关,只要获取到一个store就可以找到channel_code.
这个过程借助定时任务,每5分钟去同步2.0中的数据到CRM中,然后从2.0中拿到store,从而等到channel_code,从而在CRM中找到对应的user

CRM接入监控(2017-03-17 13:00:00)

APP将SQL,REDIS,内部服务,外部服务接入了监控系统,对于数据库的操作,内部服务的调用,外部服务的访问都能有图形化的监控.灿伟说最好将CRM也接入到监控系统,这也有助于我对CRM的整体了解,于是开始着手.

注意事项:暂时sql不要接入监控，目前监控服务端部署在业务机器上面，资源有限。 –by 栋泉

CRM相关项目及数据库

背景

1.0升级到2.0后,大部分功能都已经迁移,但是商户入网还在使用channel_code.
1.0的channel表中只有store的概念,一个store对应多个channel_code.
2.0中去掉了channel_code,新增merchant表.表示一个merchant可以对应多个store.

引发的问题

BD在帮商户入网时,依然使用channel_code.而CRM和2.0中都没有冗余channel_code,导致仅通过CRM和2.0无法完成商户入网操作.

解决方案

CRM同步2.0的merchant与store.因为channel_code在1.0中需要根据store去查询, 而2.0无法直接获取store,需要根据merchant查询.

CRM同步1.0的channel表,并将其拆分成organization,user和role三张表,同时将channel_code冗余在user表中.

只有完成数据同步后,BD在输入channel_code时,才可以在CRM中查找到相应的user,从而完成入网操作.
上述数据同步过程请参考CRM数据同步关系by王浩
